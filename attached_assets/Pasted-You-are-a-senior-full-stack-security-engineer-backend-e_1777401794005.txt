PHASE 2 — AUTHENTICATION HARDENING
==================================================

Fix authentication weaknesses.

Current problem:
The app uses PIN-based login, possibly 4 digits, and stores sessionToken/currentEmployee in localStorage. This is risky because XSS can steal tokens.

Required fixes:

1. Replace localStorage token storage with secure session handling:
   - Prefer HttpOnly, Secure, SameSite cookies for session/auth tokens.
   - Do not store sensitive tokens in localStorage or sessionStorage.
   - The frontend should not manually expose the raw token in JavaScript.
   - Update authFetch or equivalent API client to use credentials/cookies safely.

2. Implement secure cookie settings:
   - httpOnly: true
   - secure: true in production
   - sameSite: "lax" or "strict" depending on app flow
   - reasonable expiration time
   - clear cookie correctly on logout

3. Improve login security:
   - Add rate limiting specifically on /api/auth/login.
   - Add IP-based and username-based throttling if possible.
   - Add temporary lockout after repeated failed attempts.
   - Never reveal whether username or PIN/password is wrong.
   - Use a generic error message like:
     "Invalid username or password"
   - Do not return sensitive employee data after login.

4. Improve PIN/password handling:
   - If the system currently uses plain PINs, migrate to hashed secrets.
   - Use a strong password hashing method such as bcrypt/argon2.
   - Do not log PINs/passwords.
   - If the project must keep PIN login for business reasons, still hash the PIN and enforce rate limiting strongly.
   - If possible, support stronger passwords instead of 4-digit PINs.

5. Logout:
   - Invalidate server-side session if applicable.
   - Clear the HttpOnly cookie.
   - Ensure old tokens/sessions cannot be reused after logout if server-side sessions are used.

Acceptance criteria:
- No auth token exists in localStorage.
- No sensitive employee object exists in localStorage.
- Login has rate limiting.
- Failed login messages are generic.
- Password/PIN is hashed, not stored in plain text.
- Logout clears the session securely.

PHASE 4 — BOLA / OBJECT-LEVEL AUTHORIZATION
==================================================

Fix Broken Object Level Authorization.

Current risk:
A user may change an ID in the URL and access another customer, ticket, order, payment, reminder, or employee record.

Required fixes:

For every endpoint that uses an ID, verify that the authenticated user has permission to access that exact object:

- GET /api/customers/{id}
- PUT /api/customers/{id}
- DELETE /api/customers/{id}
- GET /api/tickets/{id}
- PUT /api/tickets/{id}
- PATCH /api/tickets/{id}
- DELETE /api/tickets/{id}
- POST/PATCH /api/tickets/{id}/activate
- POST/PATCH /api/tickets/{id}/deactivate
- GET/PUT/PATCH/DELETE /api/employees/{id}
- /api/orders/{id} if exists
- /api/reminders/{id} if exists
- /api/payments/{id} if exists

Implementation rules:
1. Do not just check that the user is logged in.
2. Check ownership, branch, assigned employee, company/tenant, or role.
3. If the project has employees and admins:
   - Admin can access all records.
   - Normal employee can only access records assigned to them or allowed by business rules.
4. If there is no ownership model, design a simple safe one based on the existing schema and explain it.
5. Make sure list endpoints also filter records according to the user’s permissions.

Acceptance criteria:
- User A cannot access User B’s private records by changing IDs.
- List endpoints do not return unauthorized records.
- Object-level checks are applied consistently.
- Tests or manual verification steps are provided.
